2. THE 3JUDGE TRIANGULATION ALGORITHM
This is the core mechanism that makes the system more than a gimmick.
The algorithm must produce:
agreement zones
conflict zones
blindspot exposure
combined risk score
actionable next steps
Here is the structural design.
Step 1 — Each judge produces a vector
Each lens outputs a 5dimensional vector:
V=[structure,viability,risk,narrative,evidence]
Each judge fills only the dimensions relevant to their lens; others remain null.
Step 2 — Normalize scores
All scores are normalized to a 0–1 scale.
This allows crossjudge comparison.
Step 3 — Triangulate agreement
Agreement = dimensions where all 3 judges produce similar values (within ±0.15).
This identifies consensus.
Step 4 — Triangulate conflict
Conflict = dimensions where judges diverge significantly (≥0.35 spread).
This identifies blindspot exposure.
Step 5 — Weighted synthesis
Each judge has a weight based on lens relevance to the submission type.
Example:
A medical claim → Doctor + Nutritionist + Cambodian Nurse get higher weights
A business idea → MJ + Grey + Jaque get higher weights
Weights are dynamic.
Step 6 — Combined score
S=∑(judge_weight⋅judge_vector)
This produces a multilens composite score.
Step 7 — Generate next steps
Next steps are derived from:
lowest scoring dimensions
highest conflict zones
highest risk flags
This produces actionable guidance, not generic advice.

Risk icon

The submission type suggests a better judge
A swap is not allowed when:

All evaluations from the two judges who remain
All Gauntlet stage results for those two judges
All triangulation data involving those two judges
All heatmap values derived from those two judges

Recalculate agreement
Recalculate conflict
Recalculate gaps
Recalculate risk flags
This is a partial rebuild, not a full rebuild.
Operation 4 — Update Heatmap
Only dimensions affected by the new judge are updated:
Weighted score
Coverage intensity
Conflict markers
Risk icons
Gap indicators
This is a targeted update, not a redraw.

The weighting logic is reapplied:
finalWeightj=baseWeightj×relevanceFactorj×userSelectionFactor
Then normalized.
This ensures the new triad is balanced.

Stories elevate psychological risk
Products elevate operational risk
All five feed the tier.
2. RISK SCORE CALCULATION
The system computes a composite risk score:
R=(D+F+S+C+W)
Where:
D = weighted average of risktype dimensions
F = risk flag multiplier
S = subdimension severity
C = conflict penalty
W = submissiontype weighting factor
Each component is described below.
3. COMPONENT A — DIMENSION RISK (D)
For each risk dimension:
Dd=∑(judgeWeightj⋅scorejd)
Then averaged across all risk dimensions:
D=mean(Drisk,Dclinical,Dcultural,Dpsych)
This is the baseline risk.
4. COMPONENT B — RISK FLAG MULTIPLIER (F)
If any judge score &gt; 0.7:
Add +0.15 for each flagged dimension
Add +0.05 for each additional judge flag on the same dimension
This captures severity, not just presence.
5. COMPONENT C — SUBDIMENSION SEVERITY (S)
Each subdimension has a severity rating:
Low = 0.05
Moderate = 0.10
High = 0.20
Critical = 0.30
The system sums the top three severities:
S=severity1+severity2+severity3
This prevents runaway inflation.
6. COMPONENT D — CONFLICT PENALTY (C)
If judges disagree on a risk dimension:
Mild conflict (0.35–0.45) → +0.05
Moderate conflict (0.45–0.55) → +0.10
Severe conflict (≥0.55) → +0.20
This captures uncertainty as risk.
7. COMPONENT E — SUBMISSIONTYPE WEIGHTING (W)
Each submission type has a risk multiplier:
Health Concept → +0.20
Research Concept → +0.15
Product → +0.10
Prototype → +0.10
Idea → +0.05
Story → +0.03
Claim → +0.15
This ensures context matters.
8. FINAL RISK SCORE NORMALIZATION
The raw score is normalized to 0–1:
Rfinal=min(1,R)
This becomes the risk tier input.
9. RISK TIER THRESHOLDS
The system assigns a tier:
Tier 1 — Low Risk
0.00≤R&lt;0.25
Tier 2 — Moderate Risk
0.25≤R&lt;0.50
Tier 3 — High Risk
0.50≤R&lt;0.75
Tier 4 — Critical Risk
0.75≤R≤1.00
These thresholds are fixed, not dynamic.

This identifies which judges are most relevant to the submission.

C. Conflict Potential Score

D. SubmissionType Weighting

If the submission is highrisk, riskoriented judges are boosted.
F. EvidenceAdequacy Score
If the submission is evidencedependent, evidence judges are boosted.

triadScore=0.35COV+0.25COMP+0.20TYPE+0.10RISK+0.10EVID

1. When evidence augmentation triggers
The workflow activates when any of these are true:
Evidence dimension score &lt; 0.45
Nutritionist or Doctor flags “insufficient evidence”
Subdimensions (mechanistic plausibility, citation quality, data sufficiency) show gaps
A claim is marked “strong assertion” with “no evidence”
Conflict between evidenceoriented judges (e.g., Nutritionist vs Doctor)
Trigger is explicit, not implicit.
2. Highlevel flow
Detect evidence gaps
Localize gaps to specific claims/subdimensions
Ask the user for targeted evidence
Classify and structure new evidence
Reevaluate affected dimensions
Update triangulation, heatmap, and risk tier
Log augmentation in the report
3. Step 1: Detect evidence gaps
From existing evaluation:
Use Evidence dimension + subdimensions:
Mechanistic plausibility
Citation quality
Study relevance
Data sufficiency
Confounders/bias
Mark any subdimension with:
Coverage = 0, or
Score &lt; 0.45, or
Explicit “missing data” flag
Output: list of Evidence Gap Items tied to specific claims.
json

"claimId": "C2",
"gapType": "Data sufficiency",
"severity": "High"
}
4. Step 2: Localize gaps to claims
Each gap is mapped to:
Claim ID
Claim text
Gap type (mechanism, citation, sample size, population mismatch, etc.)
Severity (Low/Moderate/High/Critical)
This produces a Claim–Evidence Gap Map.
5. Step 3: Targeted user prompts
For each gap, the system asks one precise question, not a generic “add evidence” request.
Examples:
“For Claim 2, do you have any human data supporting this effect?”
“For Claim 1, can you provide at least one peerreviewed citation?”
“For Claim 3, is there any mechanistic rationale (how it works) you can reference?”

URL
Citation text
File upload (study/report)
“No evidence available”
All responses are tied back to the claim and gap type.
6. Step 4: Evidence classification and structuring
Each new evidence item is classified as:
Evidence type (RCT, observational, case report, in vitro, animal, expert opinion, internal data, anecdote)
Population (human/animal, size, demographics if known)
Relevance (direct, indirect, weak)
Quality (high/medium/low, based on simple rules)
Structured as:
json

"type": "RCT",
"relevance": "direct",
"quality": "high",
"source": "URL or citation",
"notes": "Supports effect size in adults."
}
7. Step 5: Reevaluation logic
Only affected pieces are reevaluated:
Nutritionist rescores Evidence + subdimensions
Doctor rescores Clinical risk if evidence changes risk profile
Risk tier recalculates if new evidence reduces or increases risk
Triangulation matrix updates Evidence row
Heatmap updates Evidence row and any linked risk rows
No full rerun. Targeted recalculation only.

The system generates an Evidence Impact Summary:
Which claims moved from “unsupported” → “partially supported” → “well supported”
Which claims remain unsupported
How Evidence dimension score changed
How Risk tier changed (if at all)
Example:
“Claim 2 moved from unsupported to partially supported based on one small RCT (n=60, adults). Evidence score increased from 0.32 to 0.55. Overall risk tier remains High due to unresolved misuse potential.”

The final report gets an Evidence Augmentation section:
List of claims with added evidence
Summary of new sources
Changes in scores
Remaining gaps
Recommended next evidence steps (e.g., “Need larger human trials”, “Need mechanism clarification”)
This preserves transparency: what changed, why, and how much.

The Synthesis Brain is a dedicated module that:
Reads all judge outputs, triangulation, risk, and evidence
Identifies the most important insights and risks
Produces a single, coherent, strategic summary
Translates raw evaluation into clear guidance for the user
It does not re score. It explains and prioritizes.
2. Inputs
The Synthesis Brain consumes:
Submission object
Judge evaluations (all stages, all judges)
Triangulation result
Heatmap
Risk tier and risk drivers
Evidence augmentation status
Meta Evaluator report (if present)
User intent profile (if present)
3. Core responsibilities
Identify top 3 strengths
Identify top 3 risks
Identify top 3 leverage points (highest impact improvements)
Explain why the risk tier is what it is
Explain how evidence quality affects confidence
Connect findings back to user goals and intent
Provide a short action plan
4. Internal structure
Think of it as four passes over the data.
Signal extraction
Pulls high scoring dimensions
Pulls low scoring dimensions
Pulls high conflict areas
Pulls high risk flags
Pulls unresolved evidence gaps
Priority ranking
Ranks issues by impact on risk tier
Ranks issues by impact on viability
Ranks issues by alignment with user goals
Narrative framing
Converts ranked items into:
strengths
risks
leverage points
Chooses tone based on submission type and user intent
Action synthesis
Produces 3 to 5 concrete next steps
Ties each step to a specific dimension or judge insight

Task 7: System recommendations

Developer README for orchestration_master.json
1. Purpose
orchestration_master.json is the single source of truth for how the evaluation pipeline runs. It defines:
Module order
Module inputs and outputs
Config file locations
Error handling and audit behavior
No evaluation logic lives in code. Code reads this file and executes the pipeline accordingly.
2. File location
Recommended path:
text
/system/orchestration/orchestration_master.json
3. Structure overview
Key sections:
pipeline_order: ordered list of module ids.
modules: definition of each module, its inputs, outputs, and config files.
error_handling: how to behave on failure.
audit_trail: what to log for reproducibility.
4. How the orchestrator should be used
Load orchestration_master.json at application startup.
For a given evaluation:
Read pipeline_order.
For each module id:
Look up its definition in modules.
Resolve its inputs from previously produced outputs.
Load its config files.
Call the corresponding service or LLM prompt.
Store outputs in the database.
If any module fails:
Follow error_handling.regenerate_on_failure and fallback_module rules.
After completion:
Store all outputs and metadata defined in audit_trail.store.
5. How to add a new module
Implement the module as a service or LLM prompt.
Add a new entry under modules with:
id
inputs
outputs
config_files
description
Insert the module id into pipeline_order at the correct position.
Update any dependent modules’ inputs if needed.
No other code changes should be required if the orchestrator is implemented correctly.
6. Developer responsibilities
Do not hardcode pipeline order in code.
Do not hardcode config file paths in code.
Always read from orchestration_master.json.
Keep this file in version control and tag changes with orchestration version.
7. Quick sanity checks
Before deploying changes to orchestration_master.json:
Every id in pipeline_order exists in modules.
Every module has at least one output.
Every input of every module is produced by:
a previous module, or
the initial submission context.

1. Purpose
orchestration_master.json is the single source of truth for how the evaluation pipeline runs. It defines:
Module order
Module inputs and outputs
Config file locations
Error handling and audit behavior
No evaluation logic lives in code. Code reads this file and executes the pipeline accordingly.
2. File location
Recommended path:
text
/system/orchestration/orchestration_master.json
3. Structure overview
Key sections:
pipeline_order: ordered list of module ids.
modules: definition of each module, its inputs, outputs, and config files.
error_handling: how to behave on failure.
audit_trail: what to log for reproducibility.
4. How the orchestrator should be used
Load orchestration_master.json at application startup.
For a given evaluation:
Read pipeline_order.
For each module id:
Look up its definition in modules.
Resolve its inputs from previously produced outputs.
Load its config files.
Call the corresponding service or LLM prompt.
Store outputs in the database.
If any module fails:
Follow error_handling.regenerate_on_failure and fallback_module rules.
After completion:
Store all outputs and metadata defined in audit_trail.store.
5. How to add a new module
Implement the module as a service or LLM prompt.
Add a new entry under modules with:
id
inputs
outputs
config_files
description
Insert the module id into pipeline_order at the correct position.
Update any dependent modules’ inputs if needed.
No other code changes should be required if the orchestrator is implemented correctly.
6. Developer responsibilities
Do not hardcode pipeline order in code.
Do not hardcode config file paths in code.
Always read from orchestration_master.json.
Keep this file in version control and tag changes with orchestration version.
